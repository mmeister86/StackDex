# StackDex MVP Gap Closure Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Bring the current local-first StackDex prototype from a working vertical slice to a more complete MVP baseline.

**Architecture:** Keep the existing SwiftUI + SwiftData local-first architecture and extend the current camera/photo -> Vision OCR -> Convex lookup -> collection save flow. Prioritize UX hardening, richer result/detail presentation, and verification before adding sync/auth complexity.

**Tech Stack:** SwiftUI, SwiftData, Vision, AVFoundation, PhotoKit, URLSession, Convex HTTP query endpoint, Swift Testing, XCTest UI tests.

---

## Current Integration Findings

- The iOS client is already pointed at the production backend URL `https://backend.stackdex.de` via `STACKDEX_CONVEX_URL` and the query path `cards:lookup`.
- The current app integration is a hand-rolled HTTP POST to Convex `/api/query`; the repository does not currently show imported usage of the official Convex Swift client.
- The Xcode project inspection does not currently show a linked Swift package product for Convex, despite the intended backend direction.
- No Convex backend source or Pokewallet adapter code is present in this repository, so production integration planning depends on whether backend logic remains behind a narrow BFF boundary or shifts to direct Convex client usage.

## Open Architecture Decision

- Decide whether the iOS app should continue using the existing narrow BFF query boundary (`cards:lookup` over HTTP) or switch now to the official Convex Swift client for typed queries/mutations/actions.

## Decisions Made For Production Backend Integration

- Client strategy chosen: keep the narrow HTTP BFF boundary; iOS continues calling backend-facing lookup endpoints instead of exposing Pokewallet or direct Convex client semantics in the app.
- First backend scope chosen: implement production lookup plus card details plus pricing in the first backend pass, rather than lookup-only.
- Implication: the existing `cards:lookup` contract likely needs to evolve into a richer normalized response, or be complemented by additional backend endpoints that still preserve the BFF boundary.
- Pokewallet integration inputs available: API key and documentation are already available, so the execution plan can target the real provider contract rather than placeholders.

## Convex Monorepo Verification

- Context7 confirms that a repo-root Convex setup is standard: `package.json`, `convex.json`, and a root `convex/` folder are first-class supported project structure.
- Context7 also confirms the critical runtime constraint: Convex **queries cannot call external APIs**; external HTTP calls belong in public `action` or `internalAction` functions.
- Therefore, a live Pokewallet-backed `cards:lookup` implementation cannot remain a public Convex query if it needs on-demand provider fetches.
- Convex HTTP API supports `POST /api/query`, `POST /api/mutation`, and `POST /api/action`, so iOS can keep the narrow BFF boundary while switching only the transport endpoint from `/api/query` to `/api/action` if needed.
- Sosumi/Apple docs confirm the client side is unconstrained here: `URLSession` + `URLRequest` is the standard Apple transport for app-to-backend JSON HTTP, so changing the backend endpoint from `/api/query` to `/api/action` is not an iOS architecture risk.

## Corrected Backend Architecture Recommendation

- Recommended for live Pokewallet lookup: implement `cards:lookup` as a **public Convex action** invoked via `POST /api/action`.
- The public action should validate input, call an internal Pokewallet integration layer, normalize to `cards.lookup.v1`, and optionally write-through cache using `ctx.runMutation`.
- Keep Pokewallet-specific fetch/auth/timeout/mapping code in `convex/pokewallet.ts` behind internal functions.
- Keep versioned response shaping in a dedicated contract mapper module (for example `convex/lib/cardsLookupContract.ts`) so iOS tests and backend output stay aligned.
- Optional later optimization: add cache-first behavior with a query+action split, but do not start there unless latency/rate limits force it.

**Step 1: Write failing action tests with `convex-test`**

Cover:

- `responseSchemaVersion` mismatch request is rejected
- empty provider results return empty success envelope
- provider error propagates as action failure

**Step 2: Implement the public action**

Use a public `action`, not a query, because live Pokewallet fetches are required.

Recommended shape:

```ts
export const lookup = action({
  args: {
    recognizedTexts: v.array(v.string()),
    query: v.string(),
    maxResults: v.number(),
    responseSchemaVersion: v.string(),
    hints: v.optional(v.object({
      normalizedQuery: v.string(),
      nameTokens: v.array(v.string()),
      possibleNumbers: v.array(v.string()),
    })),
  },
  returns: v.any(),
  handler: async (ctx, args) => {
    // validate version, call internal action, map envelope
  },
});
```

**Step 3: Keep version negotiation strict**

Only `cards.lookup.v1` is accepted for this slice. Return a clear application error for other versions.
