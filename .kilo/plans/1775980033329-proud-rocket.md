# StackDex Planning Notes

## Source Reviewed
- PRD: `.docs/prd.md`
- Repo context: current repo has only the initial commit available in history.

## Product Snapshot
- Native iOS app in SwiftUI for scanning physical Pokemon cards, identifying them, and managing a local/optional synced collection.
- MVP centers on reliable scan confirmation, stack-based collection management, pricing visibility, and guest-first usage.
- Backend direction in PRD is Convex, self-hosted via Coolify, with server-side Pokewallet integration and caching.

## Implementation-Relevant Open Questions
- Exact persistence strategy on iOS for guest/local mode: SwiftData, Core Data, SQLite/GRDB, or another layer.
- Exact scan architecture split: which parts run fully on-device vs. which extracted signals are sent to the backend.
- Canonical source for pricing and card metadata freshness rules.
- Auth scope for MVP: which providers are truly in v1 vs. later.
- Merge strategy when a guest user upgrades to an account.
- Visual direction: Pokemon-inspired branding vs. neutral collector app.
- Value calculation rules: stack-level aggregate only or per-item condition-aware valuation.

## First Clarification Priority
- Decide the initial technical slice for MVP implementation, because it changes app architecture and order of work.

## Decisions Made
- MVP slice chosen: `B` - local-first iOS app, backend only for card recognition support, metadata, and pricing; auth/sync follow later.
- Local persistence chosen: `SwiftData` for guest mode and offline collection storage.
- First implementation slice chosen: `A` - real scan vertical slice from camera/photo import through analysis, backend lookup, and result confirmation.
- Backend approach chosen: `A` - set up Convex as the real BFF from the start for card search, detail, and pricing flows.
- Recognition split chosen: `A` - on-device pre-analysis with Vision/OCR, sending only extracted hints/signals to Convex for matching.
- Auth scope chosen for first release: `A` - guest mode only; login/sync stay out of the initial shipped MVP.
- First scan slice input sources chosen: `B` - ship camera and photo library import together in the first end-to-end slice.
- Top-level navigation chosen: `A` - `TabView` with `Sammlung`, `Scannen`, and `Einstellungen`, with detail/result flows pushed via `NavigationStack`.
- Save-flow metadata chosen: `A` - require only quick quantity before first save; condition stays optional and editable later.
- Duplicate handling chosen: `A` - default to `Zum Stapel hinzufügen`, while still offering `Als eigenes Exemplar speichern` before persistence.
- Uncertain-match candidate count chosen: `A` - show at most `3` ranked candidates when recognition confidence is insufficient.
- Value logic chosen: `B` - card condition already influences displayed valuation in MVP, preferably sourced directly from API pricing data when available rather than local multipliers.
- Missing condition-price fallback chosen: `A` - fall back to the general card price and clearly mark the valuation as approximate in the UI.
- Condition taxonomy chosen: `B` - use TCG-style condition levels (`Mint`, `Near Mint`, `Lightly Played`, `Moderately Played`, `Heavily Played`, `Damaged`) with localized UI labels in German and English.
- No-match recovery chosen: `A` - send users into a manual search flow prefilled with OCR hints such as card name, number, and possible set hints.
- Stack definition chosen: `A` - a stack represents one canonical card/print, containing multiple owned copies that may differ by condition.
- Stack internals chosen: `A` - each stack stores aggregated owned quantities per condition instead of persisting every physical copy as a separate record.
- Manual search availability chosen: `B` - manual search is always available in the app and also serves as the recovery path after `No Match`.
- Manual search interaction chosen: `B` - manual search uses a free-text query with optional structured hints such as card number and set.
- Price freshness chosen: `B` - server-side pricing is cached and refreshed on a bounded schedule (for MVP, roughly once per day per card rather than fetched live every time).
- External data source chosen: `A` - Pokewallet is the single canonical external source for card metadata and pricing in MVP, and the architecture is intentionally built around that API.
- Camera scan interaction chosen: `A` - users frame the card and explicitly capture a photo before on-device OCR and backend matching run.
- Scan preprocessing chosen: `B` - start with full-image automatic analysis, and only offer an optional crop/refinement step when confidence is weak.
- Scan image retention chosen: `A` - captured and imported images are used temporarily for OCR/matching and then discarded rather than stored with the collection.
- Offline behavior chosen: `B` - collection features stay usable offline, while scan and manual search surface a clear offline error and require connectivity to continue.
- Initial collection ordering chosen: `B` - the collection uses a flat list sorted by most recently added or updated stacks so new additions stay immediately visible.
- Collection discovery chosen: `B` - the collection tab includes simple local search over card name, set, and card number in MVP.
- Stack detail scope chosen: `B` - stack detail shows base card info plus the quantity breakdown per condition and simple editing for those condition buckets.
- Stack editing chosen: `B` - in stack detail, users can increase or reduce quantities per condition bucket, and buckets disappear when their quantity reaches zero.
- Stack deletion chosen: `A` - users can delete an entire stack from the detail screen, protected by a clear destructive confirmation.
- Launch tab chosen: `A` - the app opens on `Sammlung` by default rather than `Scannen` or restoring the last tab.
- Empty-state behavior chosen: `B` - an empty collection shows a primary `Karte scannen` CTA and a secondary `Manuell suchen` CTA.
- Collection structure chosen: `B` - MVP supports multiple collections/folders from the start instead of a single master collection.
- Multi-collection entry chosen: `B` - the `Sammlung` tab opens the last-used or pinned collection directly, with a clear switcher for moving between collections.
- Collection management chosen: `B` - users can explicitly create, rename, and delete collections in MVP.
- Add-target behavior chosen: `B` - new cards default into the currently active collection, but the save flow exposes a clear way to change the target collection before persistence.
- Scan-tab target visibility chosen: `B` - the global `Scannen` tab clearly shows the current target collection at the top and allows switching it before saving.
- Missing-target recovery chosen: `B` - if no active collection exists yet, or the active collection was deleted, the app immediately prompts the user to create or choose a collection and then uses it as the active target.
- Default-collection behavior chosen: `B` - users can explicitly define a pinned/default collection instead of relying only on the last active collection.
- Non-empty collection deletion chosen: `B` - a non-empty collection can be deleted directly in MVP, but only behind a strong destructive confirmation.
- Deleted-default recovery chosen: `A` - when the deleted collection was also the pinned/default one, the app automatically falls back to the last active remaining collection; if none remain, it shows the create-or-choose flow.
- Collection inventory model chosen: `B` - collections are independent inventories, so the same canonical card may exist separately in multiple collections.
- Cross-collection correction chosen: `C` - stacks can both be moved and copied between collections in MVP.
- Move/copy granularity chosen: `A` - move and copy actions operate on the complete stack only, not on individual condition buckets.
- Cross-collection landing conflict chosen: `B` - when a moved or copied stack lands in a collection that already contains the same canonical card, the destination collection may still keep a separate parallel stack instead of auto-merging.
- In-collection add behavior chosen: `A` - normal scan/manual add flows still default to `Zum Stapel hinzufügen`; separate parallel stacks are created only through explicit user choice or via move/copy flows.
- Overview scope chosen: `B` - MVP includes a simple cross-collection overview for total value and total card count across all collections.
- Global overview placement chosen: `B` - the cross-collection overview lives as an `Alle Sammlungen` entry inside the collection switcher alongside individual collections.
- Collection metadata chosen: `B` - each collection has a name plus an optional short description in MVP.
- Global overview visual direction noted: `Alle Sammlungen` should present each collection like a visible card stack, with one top card preview and a few key metrics shown underneath.
- Collection preview source chosen: `A` - each collection tile in `Alle Sammlungen` uses the newest or topmost real card from that collection as the visible preview image.
- Collection tile metrics chosen: `A` - each collection tile shows the collection name, total value, and total card count beneath the visible card-stack preview.
- Empty collection representation chosen: `B` - empty collections still appear in `Alle Sammlungen` with a neutral placeholder stack/cover while keeping the same metric layout.
- Collection-tile tap behavior chosen: `A` - tapping a collection tile in `Alle Sammlungen` opens that collection directly and also makes it the active collection.
- Global cross-collection search chosen: `C` - a true search across all collections is deferred beyond MVP, though the architecture should leave room to add it later.
- Default-target constraint chosen: `A` - `Alle Sammlungen` cannot be used as the pinned/default collection because it is an overview, not a valid save target.
- Permission timing chosen: `B` - camera and photo access are requested just-in-time when users first enter the relevant action rather than up front at app launch.
- Photo-import access chosen: `B` - MVP needs broader photo-library access beyond the system picker alone, implying an in-app browsing/import experience rather than only `PhotosPicker`.
- Photo-import UX chosen: `A` - the `Scannen` tab includes a simple built-in photo browser focused on quickly selecting recent/relevant images rather than a full albums-style library app.
- Photo-browser scope chosen: `A` - the in-app photo browser is limited to a fast recent-images strip/grid in the scan flow rather than full-library browsing.
- Scan-tab layout chosen: `A` - the `Scannen` tab uses the camera as the main surface with a compact recent-photos area coexisting in the same screen.
- Recognition start chosen: `A` - after capture or recent-photo selection, recognition starts automatically without an extra review step.
- Denied-permission recovery chosen: `B` - permission denial states explain the issue in-context, offer `Einstellungen öffnen`, and provide an alternative path where possible such as photo import or manual search.
- Settings scope chosen: `B` - the `Einstellungen` tab includes permissions/app info as well as collection management and pinned/default collection controls.
- Language behavior chosen: `A` - the app follows the system language in MVP instead of offering an in-app language switch.
- Visual direction chosen: `B` - the MVP should feel TCG-elegant: collection-forward and recognizable for card collectors, but still clean, modern, and not overly playful.
- Price-indicator scope chosen: `A` - MVP surfaces current values but does not yet add explicit trend arrows, change deltas, or market-status indicators in collection lists.
- Recognition confidence messaging chosen: `B` - the app should use gentle human-readable uncertainty hints like `Treffer prüfen` rather than exposing technical confidence values.
- Sharing/export scope chosen: `A` - MVP does not include collection sharing or export, keeping the first release focused on private inventory management.
- Offline cache scope chosen: `B` - stored cards cache core metadata plus the card image locally so collection browsing and detail views remain useful offline.
- Manual price refresh chosen: `A` - MVP does not expose manual refresh actions for valuations; pricing updates happen automatically through the scheduled backend refresh model.
- Value freshness visibility chosen: `A` - pricing freshness remains internal in MVP and is not surfaced as a user-visible timestamp or status in the UI.
- Onboarding scope chosen: `A` - MVP relies on empty states and in-context hints instead of a dedicated tutorial or intro flow.
- Collection switcher ordering chosen: `A` - collection entries in the switcher are primarily ordered by most recently used rather than alphabetically or by a fixed pinned-first position.
- Pinned-collection switcher behavior chosen: `A` - the pinned/default collection stays in recency order and is distinguished with a badge instead of occupying a fixed top position.
- Missing-price handling chosen: `A` - cards without a usable Pokewallet price still appear normally in the collection, but their value remains unknown/empty and is not fabricated.
- Partial-valuation totals chosen: `B` - collection totals remain visible when some cards lack prices, but the UI should indicate that the displayed sum is incomplete.
- Collection row density chosen: `A` - the main stack list should stay compact, with richer information deferred to the detail screen.
- Compact row fields chosen: `A` - stack rows must at least show preview image, card name, and total quantity.
- Print differentiation chosen: `A` - compact collection rows use a small secondary line with set and/or card number so repeated card names remain distinguishable.
- Collection row interaction chosen: `A` - collection rows stay tap-to-open only in MVP and do not expose direct quick actions from the list.
- Planning fallback rule chosen: for any remaining minor product or UX decisions not explicitly answered, use the previously recommended option by default.

## Doc Validation Notes (Sosumi + Context7)
- `SwiftData` documentation confirms the planned local-first model is a good fit for `@Model` classes, relationships, and a `ModelContainer`-based app setup. No doc-level blocker found for the planned collection -> stack -> condition-bucket model shape.
- Apple `PHPhotoLibrary` docs confirm that the broader in-app recent-photo browser requires explicit photo-library authorization handling, including support for `.limited` access and `presentLimitedLibraryPicker(from:)` for recovery/expansion.
- Apple `PhotosPicker` docs confirm that the system picker remains a viable lower-friction fallback, but it doesn't replace the need for PhotoKit authorization if the app wants an embedded recent-images browser inside `Scannen`.
- Apple `AVCaptureDevice.RotationCoordinator` docs validate the recommended custom-camera path for iOS 17+, including separate horizon-level rotation angles for capture and preview.
- Apple `VNRecognizeTextRequest` docs validate the OCR-based recognition strategy and reinforce that language configuration, recognition level, and optional language correction should be treated as first-class tuning points in the scan pipeline.
- Context7 confirms Convex supports self-hosting and also provides an official Swift client for iOS/macOS that can call queries, mutations, actions, and auth flows directly.
- Context7 also confirms Convex exposes an HTTP API (`/api/query`, `/api/mutation`, `/api/action`), but the Swift client is the more natural primary integration path for this app unless a narrower HTTP boundary is intentionally preferred.

## Plan Adjustments After Doc Review
- Backend integration planning should assume `Convex Swift client first`, with HTTP endpoints reserved for edge cases or deliberately server-only flows, instead of treating generic HTTP access as the default mobile integration.
- Photo import planning must explicitly include limited-library UX and recovery behavior as a required part of the embedded recent-photos feature, not as an optional polish item.
- OCR implementation planning should include configurable language/recognition settings rather than assuming a single fixed `VNRecognizeTextRequest` setup.
