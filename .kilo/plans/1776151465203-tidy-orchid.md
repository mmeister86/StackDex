# Scan-OCR Reliability Implementation Plan

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Make card recognition robust enough that scan photos produce stable card-name / card-number hints instead of noisy full-card gibberish, while reshaping the Scan tab into a dedicated scanner UI with reliable focus behavior.

**Architecture:** Replace the current mixed scroll-page scan screen with a dedicated scanner surface that prioritizes capture first: full scanner canvas, top-right recent-photo shortcut, centered card frame, primary capture CTA, and explicit focus control. Under the hood, keep the still-photo capture flow but turn it into a staged pipeline: normalize image input and orientation, detect and rectify the card region, run trading-card-tuned OCR passes on specific regions, then build structured lookup hints and narrower fallback queries. Pair that with camera focus management and scan telemetry so regressions are visible.

**Tech Stack:** SwiftUI, AVFoundation, PhotoKit, Vision, CoreImage, OSLog, Swift Testing, XCTest UI tests.

---

## Current Findings

- `StackDex/Services/ScanPipelineService.swift:60` runs `VNRecognizeTextRequest` directly on the raw full-frame image, with no card localization, no perspective correction, no region-of-interest pass, and no confidence filtering.
- `StackDex/Services/ScanPipelineService.swift:32` uses mixed `de-DE` / `en-US` recognition plus `usesLanguageCorrection = true`, which is better for prose than for card names, set codes, and collector numbers.
- `StackDex/Services/ScanPipelineService.swift:74` keeps only the top candidate for every observation and joins everything into one broad query; tiny rules text and background text can dominate the useful signal.
- `StackDex/Services/CameraCaptureService.swift:81` captures with `.balanced` photo prioritization, which is a poor default for OCR.
- `StackDex/ScanTabView.swift:116` shows a plain preview with no scan guide, no ROI hint, and no framing feedback.
- `StackDex/Services/RecentPhotoLibraryService.swift:85` requests image data and orientation but drops the returned orientation when reconstructing the `UIImage`.
- There are currently no OCR pipeline tests, no scan telemetry, and no UI path that exercises camera/import recognition in tests.

## Implementation Direction

- Optimize for the current trading-card flow already implied by the app and tests.
- Prefer a smaller, high-confidence structured query over shipping the whole OCR dump to the backend.
- Keep the existing capture -> OCR -> lookup -> save architecture; do not replace it with live `DataScannerViewController` in this batch.
- Add low-risk instrumentation first so future tuning has evidence.
- Use a dedicated scanner layout as the primary Scan-tab structure: camera-first, scroll-free during capture, with result/manual search content moved behind the capture-first presentation rather than competing with it on the same screen.
- Implement both continuous autofocus for the framed card area and tap-to-focus with visible feedback; the scanner must not depend on fixed-focus behavior.
- The app already targets iOS 17.6, so the implementation can standardize on `AVCaptureDevice.RotationCoordinator` directly instead of carrying a legacy pre-iOS-17 orientation path.
- Tap-to-focus must follow AVFoundation’s documented flow: convert the tapped preview point with `AVCaptureVideoPreviewLayer.captureDevicePointConverted(fromLayerPoint:)`, gate on support checks, lock the device for configuration, set both focus and exposure points of interest, and then trigger the appropriate focus/exposure modes.

### Task 3: Detect, Rectify, And Crop The Card Before OCR

**Files:**
- Create: `StackDex/Services/CardRegionDetector.swift`
- Modify: `StackDex/Services/ScanImagePreprocessor.swift`
- Modify: `StackDex/Services/ScanPipelineService.swift`

**Step 1: Detect the dominant card/document quad**

Use `VNDetectDocumentSegmentationRequest` on the normalized image to find the card bounds. Keep a fallback path to the full image if no stable quad is found.

**Step 2: Perspective-correct the detected card**

Use `CIPerspectiveCorrection` with the detected `VNRectangleObservation` corners to generate a rectified card image before OCR.

**Step 3: Crop ROI bands for OCR passes**

Define explicit card regions after rectification, for example:
- `nameBand`: top ~22% of the card
- `numberBand`: bottom ~18% of the card
- `fullCardFallback`: full rectified image with tighter filters

**Step 4: Add failing then passing tests around ROI extraction**

Tests should assert:
- the detector falls back cleanly when no rectangle is found
- perspective correction returns a usable upright image
- OCR on the name/number bands is more stable than OCR on the full noisy frame

### Task 4: Replace Single-Pass OCR With Trading-Card-Tuned OCR

**Files:**
- Modify: `StackDex/Services/ScanPipelineService.swift`
- Create: `StackDex/Services/ScanQueryBuilder.swift`
- Test: `StackDexTests/VisionScanPipelineServiceTests.swift`
- Test: `StackDexTests/ScanQueryBuilderTests.swift`

**Step 1: Add multiple OCR pass presets**

Use separate request settings instead of one default:

```swift
OCRRequestSettings(
    recognitionLevel: .accurate,
    recognitionLanguages: ["en-US", "de-DE"],
    usesLanguageCorrection: false,
    minimumTextHeight: 0.02
)
```

Tune per band:
- `nameBand`: accurate, language correction off, medium minimum text height
- `numberBand`: accurate, language correction off, very low minimum text height
- `fullCardFallback`: accurate or fast depending on latency, but still correction off

**Step 2: Keep multiple candidates and confidence values**

For each `VNRecognizedTextObservation`, keep the best 2-3 candidates and discard low-confidence outputs. Do not immediately flatten everything into one string.

**Step 3: Parse structured fields before building lookup hints**

`ScanQueryBuilder` should:
- deduplicate OCR tokens
- prefer top-band text for the candidate name
- prefer bottom-band numeric tokens for collector numbers
- normalize common OCR confusions in numeric contexts (`O -> 0`, `I/l -> 1`, `S -> 5`)
- build a compact `normalizedQuery` from the preferred fields only

**Step 4: Keep a controlled fallback**

If structured extraction is weak, fall back to a trimmed ranked token list instead of the full OCR dump.

**Step 5: Update tests to verify the new contract**

The passing state should confirm that noisy body text is excluded from the top-level query when better name/number fields exist.

### Task 5: Harden Lookup Handoff And Empty-Result Recovery

**Files:**
- Modify: `StackDex/Services/CardLookupService.swift`
- Modify: `StackDex/ScanTabView.swift`
- Test: `StackDexTests/ConvexCardLookupServiceTests.swift`
- Test: `StackDexTests/ScanQueryBuilderTests.swift`

**Step 1: Send structured hints first**

Keep sending `recognizedTexts`, but make the primary `query` field favor the structured query from the new builder instead of `recognizedTexts.joined(separator: " ")`.

**Step 2: Add a progressive fallback strategy**

If the first lookup returns an empty success, retry in-process with a narrower alternate query order, for example:
1. `name + number`
2. `name only`
3. ranked OCR tokens

Keep this retry loop bounded to 2-3 attempts.

**Step 3: Preserve current backend contract compatibility**

Do not break the existing request body shape; enrich the content while keeping `recognizedTexts`, `query`, and `hints` present.

**Step 4: Update tests for empty-success fallback**

Adjust the current “empty success means no fallback” behavior if the new retry policy lives above the raw transport layer. Keep transport tests focused on HTTP parsing; add separate tests for the higher-level scan lookup retry behavior.

### Task 6: Improve Capture UX For OCR Quality

> **Design correction after visual review:** the first scanner-shell implementation proved too busy and still too close to a dashboard. The next implementation must follow the provided sketch much more literally: minimal chrome, scanner-first, and no competing secondary content during the default capture state.

**Files:**
- Modify: `StackDex/Services/CameraCaptureService.swift`
- Modify: `StackDex/ScanTabView.swift`
- Create: `StackDex/ScannerFocusIndicatorView.swift`
- Modify: `StackDex/Services/RecentPhotoLibraryService.swift`
- Test: `StackDexUITests/ScanFlowUITests.swift`

**Step 1: Prefer quality over speed for scan photos**

Change the scan capture settings to prioritize OCR quality instead of balanced capture latency.

**Step 2: Rebuild the Scan tab as a dedicated scanner surface**

Use the user-provided schema as the layout contract:
- camera canvas as the primary body
- recent-photos shortcut anchored top-right inside the scanner chrome
- centered portrait card frame
- primary `Aufnehmen` CTA anchored near the bottom
- capture-first presentation with secondary scan results/search UI de-emphasized until after capture
- post-capture information presented in a bottom sheet so the scanner remains visible behind it

Do not keep the current large vertical scroll layout as the default capture experience.

Explicit visual constraints for the new implementation:
- remove nonessential labels and chips from the live camera surface in the default state
- do not show a permanent status badge like “Scanner / Kamera zuerst, Treffer danach” over the preview
- do not show the target collection chip inside the camera shell during idle capture
- do not show a large result/manual-search panel under the scanner while idle
- do not show the focus reticle unless the user has just tapped to focus
- keep the recent-photo shortcut visually small and quiet, closer to the simple square shortcut in the sketch
- the scanner should read as one dominant surface, not as a stacked page with multiple competing cards

Default-state composition should be:
1. scanner surface
2. card frame
3. small recent-photo shortcut
4. primary capture button

Everything else should be hidden, collapsed, or moved to a secondary state.

**Step 3: Add a visible scan guide overlay**

In `ScanTabView`, add a centered card-shaped guide aligned to the new dedicated layout, but keep the overlay minimal.

Specific guidance:
- prefer the dashed frame itself over extra pill labels like “Scan-Rahmen”
- helper copy should be short and secondary, and should not dominate the camera surface
- if helper text is needed, place it subtly outside the frame hierarchy rather than turning it into another floating badge

**Step 4: Add focus behavior and feedback**

Implement two focus paths in `CameraCaptureService`:
- continuous autofocus / auto exposure for the active card area by default
- tap-to-focus + tap-to-expose on the preview, mapped from view coordinates into camera coordinates

Also add visible focus feedback in the preview layer, for example a transient focus reticle/indicator rendered by SwiftUI over the tapped location.

If the device supports it, also bias autofocus to the center frame so the card area is favored before manual taps.

Visual constraint:
- the focus indicator must be transient and absent in the idle state; it should never sit permanently in the scanner preview as decorative chrome

Implementation notes from Apple docs:
- use `AVCaptureVideoPreviewLayer.captureDevicePointConverted(fromLayerPoint:)` for coordinate conversion instead of manual math
- check `isFocusPointOfInterestSupported` / exposure support before setting points
- call `lockForConfiguration()` / `unlockForConfiguration()` around focus and exposure updates
- setting `focusPointOfInterest` alone is not enough; follow it by setting `focusMode`
- setting `exposurePointOfInterest` alone is not enough; follow it by setting `exposureMode`

**Step 5: Add rotation-safe preview/capture plumbing**

Introduce `AVCaptureDevice.RotationCoordinator` in `CameraCaptureService` / `CameraPreviewView` so the preview and captured photo stay aligned with OCR expectations. Because the deployment target is iOS 17.6, this can be the primary path rather than an optional branch.

**Step 6: Keep import and camera paths behaviorally aligned**

Both captured and imported images should flow through the same normalized preprocessing path so tuning applies uniformly.

**Step 6b: Move results and manual search out of the default capture state**

The sketch implies a scanner-first mode, not a scanner-plus-form dashboard. Reflect that in the interaction model:
- keep the default scanner state visually focused on capture only
- reveal results, save controls, and manual search only after a capture/import/no-match event in a bottom sheet
- when there is no active result, avoid rendering a large empty “Ergebnis” card below the scanner
- manual search remains available, but not as a permanently competing block in the idle scanner layout

Bottom-sheet requirements:
- scanner preview remains visible behind the sheet
- sheet supports at least a compact and expanded detent
- no-match state, result selection, save controls, and manual search live inside the sheet rather than in the base scanner layout

**Step 6a: Handle session interruptions and temporary camera loss**

Add explicit handling for `AVCaptureSessionWasInterrupted` / `AVCaptureSessionInterruptionEnded` so the dedicated scanner surface does not look frozen if a phone call, multitasking condition, or camera handoff interrupts capture. Show a clear inline scanner state rather than silently failing.

**Step 7: Add a smoke-level UI test for the scanner shell**

Do not try to automate live camera OCR. Cover the visible UX contract instead:
- dedicated scanner surface renders
- recent-photo shortcut is visible in the scanner shell
- capture CTA remains reachable in the expected states
- default idle scanner state stays visually minimal and does not render the old large result/dashboard composition
- guidance copy and no-match/result messaging still behave after a mocked lookup path
- scanner controls satisfy touch-target expectations and do not compete visually with the camera content

## Expected Outcome

- The scan pipeline stops treating the entire card as a document blob and instead extracts card-like fields deliberately.
- OCR output becomes inspectable, confidence-aware, and much less sensitive to flavor text/background noise.
- Lookup requests become narrower and more recoverable when the first pass is imperfect.
- The UI becomes a dedicated scanner experience that matches the provided schema and gives users a better chance to capture OCR-friendly images.
- The default scanner state becomes visually minimal and much closer to the provided sketch, instead of looking like a dense dashboard.
- Camera behavior actively supports scanning through autofocus, tap-to-focus, and visible focus feedback.
